# ADR-0003: Remove `adr` from RalplanState

**Date:** 2026-05-14  
**Status:** Accepted

## Context

During code review of the `ard-artifact` branch, a latent bug was identified in the state persistence logic:

- `RalplanState.adr` was typed as `ADR` interface which contains methods (`addEntry`, `approve`, `reject`, `toMarkdown`)
- `JSON.stringify()` cannot serialize functions — methods would be lost on persist
- Additionally, `state.adr` was **never set** anywhere in the extension code — it was dead code

## Decision

Remove `adr` from:
1. `PersistedState` interface in `index.ts`
2. `RalplanState` interface in `state.ts`
3. All persist/restore logic in `index.ts`

The ADR content lives in the plan markdown file (generated by AI during planning), not in runtime state. This is the single source of truth.

## Consequences

- **Removed:** Potential serialization bug
- **Removed:** Dead code that added complexity
- **Kept:** `createADR()` in `adr.ts` — exported for potential future use by tests
- **Kept:** `listWorktrees()`, `cleanupWorktree()`, `validateWorktree()` in `worktree.ts` — exported for tests

## Rationale

If ADR functionality needs to be restored later:
1. Parse the plan markdown file for ADR content (existing files remain valid)
2. Or add a new ADR object with proper serialization strategy (only persist `entries[]`, recreate methods on load)

## Alternatives Considered

| Option | Why Not Chosen |
|--------|----------------|
| Persist only `adr.entries` | Marginal benefit — `adr` is never set anyway |
| Keep with runtime-only flag | Adds complexity for no current use case |
